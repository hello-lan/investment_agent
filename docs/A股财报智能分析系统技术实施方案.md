## A股财报智能分析系统 - 完整技术实施方案

> 版本：1.0  
> 角色：CTO + 产品经理  
> 基于现有 `a-share-financial-forensic` skill 及前期讨论整合

---

## 一、项目背景与目标

### 1.1 现状
- 已实现：PDF → Markdown 转换（docling）、基础爬虫（巨潮资讯网）、一份成熟的财报排雷 skill（含38条规则、三表联动分析流程、标准化报告模板）。
- 待解决问题：
  - Markdown 质量仍有错漏，需质检与轻修复。
  - 财报文本量大，直接喂 skill 成本高、效果差。
  - 附注信息必须从原文提取并与报表项对齐，且可用第三方数据（akshare）交叉验证。
  - 需要支持跨年分析、多源数据（研报、问答等）融合。
  - 分析时需决定 RAG 与直接上下文的策略，以及知识库设计。

### 1.2 目标
构建**生产级 A股财报智能分析系统**，以现有 skill 为核心分析引擎，增加预处理、结构化提取、知识库、验证与修复层，实现：
- 高质量、低成本、可解释的财报分析。
- 支持单年、跨年、多公司比较、附注明细挖掘。
- 可集成外部数据验证，提升结论置信度。
- 最终输出结构化疑点排查报告（与现有 skill 模板兼容）。

## 二、整体架构

```
[数据采集层]
   ├─ 爬虫 (巨潮资讯网) → PDF
   ├─ akshare (三大报表数据、行业对比)
   └─ 外部文档 (研报、问答) → 文本

            ↓

[解析与质检层]
   ├─ docling → 原始 Markdown
   ├─ 质量检查器 (L1-L3) → 质量报告 + 轻修复
   └─ 附注抽取 & 与报表项对齐模块

            ↓

[结构化提取层]
   ├─ 标题分割 + 表格解析
   ├─ 规则抽取器 (关键财务指标)
   └─ 优先级过滤 (P0-P4)

            ↓

[存储层]
   ├─ 时序结构化 DB (DuckDB)
   │    └─ 指标表、附注映射表、第三方校验表
   ├─ 文档向量库 (LanceDB)
   │    └─ 语义块 (文本/表格) + 元数据
   └─ 原始 Markdown 归档

            ↓

[分析编排层]
   ├─ Context Builder (聚合多年度、外部数据)
   ├─ RAG 检索器 (按需补充细节)
   └─ 工具函数库 (供 skill 调用)

            ↓

[分析执行层]
   └─ a-share-financial-forensic skill (原有)
        ├─ 接收 Context + 用户问题
        ├─ 调用工具函数 (获取指标、检索原文)
        └─ 输出结构化报告

            ↓

[输出与反馈]
   ├─ Markdown 报告
   ├─ 数据质量标注
   └─ 用户可追问
```

## 三、模块详细设计

### 3.1 数据采集层

#### 3.1.1 财报PDF采集
- 使用已开发的 `CninfoSpider`（修复 URL 拼接及错误处理）。
- 按公司+年份下载年报、半年报、季报。
- 存储原始 PDF 至 `raw_pdfs/{stock_code}/{year}.pdf`。

#### 3.1.2 第三方数据验证 (akshare)
- 调用 `ak.stock_financial_abstract` 或 `ak.stock_financial_report_sina` 等接口获取三大报表关键数据（营收、净利润、经营现金流、资产总计等）。
- 存为 `validation/{stock_code}/{year}_akshare.json`。
- **用途**：
  - 交叉验证 PDF 解析出的数值（例如应收/营收比例、毛利率）。
  - 当 PDF 解析明显错误时，可降级使用 akshare 数据并标注“数据来源：第三方”。
  - 用于计算行业均值（横向对比）。

#### 3.1.3 外部文档采集（可选）
- 投资者问答：巨潮资讯网 “互动易” 爬虫。
- 研报：存储为文本或 PDF，后续处理与年报类似（但仅做语义检索，不提取结构化）。

### 3.2 解析与质检层

#### 3.2.1 质量检查器（三层）
| 层级 | 检查项 | 处理方法 |
|------|--------|----------|
| L1 | Markdown 是否包含必要章节标题（资产负债表、利润表、现金流量表、附注） | 缺失则标记“低质量”，仍继续但置信度下调 |
| L2 | 抽取营业收入、净利润、经营现金流，与 akshare 对比，偏差 >20% 则告警 | 记录差异，输出警告，以 akshare 为准并注明 |
| L3 | 表格列数一致性：检测每个表格行字段数，若某行明显少列，尝试用前一行补充或删除 | 自动修复并记录变更日志 |

**实现**：独立 Python 模块 `markdown_qa.py`，不依赖 LLM，速度 <0.5秒/年报。

#### 3.2.2 附注抽取与报表项对齐（关键创新）
**目标**：从附注中提取每个科目（应收账款、存货、商誉…）的明细结构，并确认其“归属于母公司”还是“合并”，与报表中的数值匹配。

**方案**：
1. **识别附注块**：正则匹配 `^#+ 附注` 或 `^#+ 注释`。
2. **解析每个附注子项**：例如 `## 附注5 应收账款` 下通常包含：
   - 按账龄列示
   - 按客户类别列示
   - 坏账准备变动
   - 期末余额/期初余额
3. **结构化存储**：
   - 表：`disclosures`
     - `stock_code`, `year`, `item_name` (如“应收账款”), `sub_item` (如“1年以内”), `value`, `unit`, `is_consolidated`。
   - 同时记录原文位置（块ID），供追溯。
4. **对齐校验**：
   - 从附注中累加明细（如“1年以内+1-2年+…”）应等于报表中“应收账款”余额。若差异超过阈值，标记为“附注不一致”。
   - 不一致时，系统优先使用报表数值（来自三大报表块），但输出分析时提示用户“附注与报表存在差异，可能影响账龄分析”。

**技术选型**：用正则 + 简单规则（因为附注格式相对固定），无需 LLM。若遇到复杂无规则附注，降级为整块保留供 RAG 检索。

### 3.3 结构化提取层

#### 3.3.1 分割策略（语义块）
- 基于 Markdown 标题层级（`#`, `##`, `###`）切分。
- 每个块包含：
  - `type`: 表格 / 文本 / 标题
  - `heading_path`: 父标题链
  - `tokens`: 预估 token 数（<1500）
  - `priority`: P0-P4 (根据标题关键词判断)
- 表格块单独存储，同时保留表格前后的文本（上下文）。

#### 3.3.2 关键指标规则抽取
使用正则从文本和表格中抽取以下指标（存入结构化 DB）：
- 营收、营业利润、净利润、扣非净利润
- 经营现金流净额、投资现金流、筹资现金流
- 资产总计、负债合计、所有者权益
- 货币资金、应收账款、存货、商誉、在建工程
- 毛利率、净利率、ROE、资产负债率
- 应收账款周转率、存货周转率

**抽取方法**：
- 优先从**财务摘要块**或**主要会计数据**表格中匹配。
- 若无，从利润表/资产负债表表格中按行匹配常见名称（支持模糊匹配：如“营业收入”、“营业总收入”）。
- 与 akshare 数据比对，若偏差过大，采用第三方数据并记录来源。

#### 3.3.3 优先级过滤
- P0 (必须保留)：三大报表数据、审计意见、核心财务指标。
- P1 (重要保留)：管理层讨论中的业绩变动原因、风险提示、未来展望。
- P2 (保留)：重大关联交易、对外担保、诉讼。
- P3 (可裁剪)：会计政策、税项、员工情况、环境信息（除非用户要求）。
- P4 (丢弃)：重复页眉页脚、目录、空白。

实施：配置关键词列表，根据标题过滤。被裁剪的内容不进入后续分析，但原始 Markdown 完整存档。

### 3.4 存储层设计

#### 3.4.1 时序结构化数据库 (DuckDB)
- **表 `financial_indicators`**：
  ```sql
  CREATE TABLE financial_indicators (
      stock_code VARCHAR,
      report_date DATE,
      indicator_name VARCHAR,
      value DECIMAL,
      unit VARCHAR,
      source VARCHAR,  -- 'pdf_parsed', 'akshare', 'manual'
      confidence FLOAT, -- 0-1
      PRIMARY KEY (stock_code, report_date, indicator_name)
  );
  ```
- **表 `disclosures`**（附注明细）：
  ```sql
  CREATE TABLE disclosures (
      stock_code VARCHAR,
      report_date DATE,
      item_name VARCHAR,      -- '应收账款'
      sub_item VARCHAR,       -- '1年以内'
      value DECIMAL,
      unit VARCHAR,
      original_chunk_id VARCHAR -- 关联向量库中的块ID
  );
  ```
- **表 `validation_log`**：记录 PDF 解析值与 akshare 的比对结果。

#### 3.4.2 文档向量库 (LanceDB)
- 每个语义块（P0-P2级别）生成 embedding（使用 `BAAI/bge-large-zh-v1.5` 或 `text-embedding-3-small`）。
- 元数据包含：`stock_code`, `year`, `block_type`, `heading`, `priority`, `chunk_id`。
- 检索时支持**按公司+年份过滤**，避免跨公司干扰。

#### 3.4.3 原始 Markdown 归档
- 按公司+年份存储，用于人工复核和回溯。

### 3.5 分析编排层

#### 3.5.1 Context Builder
**作用**：根据用户问题动态构建发送给 skill 的上下文，控制 token 成本。

- **单年基础分析**：Context = P0 指标表 + 审计意见 + 最近一年管理层讨论摘要（P1）。
- **跨年趋势分析**：Context = 近5年关键指标趋势表（结构化输出）+ 异常年份的原始块（通过 RAG 检索）。
- **附注深度分析**（如询问“应收账款账龄结构是否恶化”）：Context = 结构化抽取的附注明细 + 对比上年数据。

**实现**：一个 Python 函数 `build_context(company, years, question_type)`，返回一个字典，包含 `table_markdown`, `summary_text`, `rag_query_hint`。

#### 3.5.2 RAG 检索器
- 使用 LanceDB，查询时根据问题 embedding 检索最相关的 top-k 个块（默认 k=5）。
- 支持**硬过滤**：仅检索指定公司+年份范围内的块。
- 为降低 token 消耗，检索结果会按优先级排序，且自动截断超过 3000 tokens 的内容。

#### 3.5.3 工具函数库（供 skill 调用）
在 skill 的 `allowed_tools` 中声明以下函数（以 MCP 或自定义 API 形式提供）：

| 工具名 | 功能 | 示例返回 |
|--------|------|----------|
| `get_financial_table(stock, years, metrics)` | 获取指标时序表 | markdown 表格 |
| `get_disclosure_detail(stock, year, item)` | 获取附注明细 | markdown 列表 |
| `search_disclosure(stock, keyword, years)` | 在附注中搜索关键词 | 文本块 |
| `compare_with_peers(stock, metric, year)` | 与行业平均对比 | 数值 + 百分位 |
| `check_validation(stock, year, indicator)` | 检查 PDF vs akshare 差异 | 差异报告 |

### 3.6 分析执行层（集成现有 skill）

#### 3.6.1 修改 `a-share-financial-forensic` skill
现有 skill 已经包含完整的工作流程和规则库。需要做以下适配：

1. **输入扩展**：
   - 默认接收 `Company`, `ReportYear`, 以及由 Context Builder 生成的 `preprocessed_data`（包含结构化指标、附注摘要等）。
   - 若用户只提供原始 Markdown，skill 会调用本地工具函数先进行预处理。

2. **工具集成**：
   - 在 skill 中增加 `use_tool` 指令，调用上面定义的工具函数。
   - 例如在第三阶段“现金流量表分析”时，可调用 `get_financial_table` 获取5年经营现金流数据。

3. **规则库数据化**：
   - 将 38 条规则改写成可在代码中自动检查的表达式（例如 R001: `应收增速 - 营收增速 > 10%`）。
   - 规则引擎读取结构化 DB 中的指标值，自动标记触发规则，无需 LLM 逐条判断。
   - LLM 仅负责解释规则触发原因、生成风险推断。

4. **附注分析增强**：
   - 当规则涉及附注内容（如 R003 长账龄应收账款大幅增长），skill 调用 `get_disclosure_detail` 获取账龄数据，再结合规则判断。

5. **输出兼容**：
   - 最终输出仍遵循原有报告模板，但增加一个 `data_quality` 段落，说明哪些数据来自第三方、哪些存在解析误差。

#### 3.6.2 Skill 部署方式
- 作为 Claude Code skill（`.claude/skills/a-share-financial-forensic/SKILL.md`）。
- 或作为 MCP server，供 Claude Desktop 调用。

### 3.7 多源数据与跨年分析实现

#### 跨年分析流程（示例）
用户提问：“分析茅台过去5年毛利率变化趋势，并判断是否有造假嫌疑。”

1. Context Builder 调用 `get_financial_table('600519', [2020-2024], ['毛利率'])` → 得到 markdown 表格。
2. 同时 RAG 检索关键词“毛利率”+“解释”近5年的管理层讨论块。
3. 构建最终上下文：`表格 + top-3 文本块 + 规则库相关规则说明`。
4. Skill 执行：
   - 规则引擎检测到毛利率连续上升且增幅>10%（R032），标记可疑。
   - LLM 根据检索到的管理层解释（如“产品结构优化”）判断合理性，若解释空洞则给出高风险信号。
5. 输出报告包含趋势图（文字描述）和风险推断。

#### 外部文档融合（研报/问答）
- 与年报分开存储（不同集合）。检索时用户可选择是否包含。
- 若用户提供额外文档（上传文件），系统先分割、embedding后临时加入当前会话的向量库（不持久化），并允许 skill 检索。

## 五、风险与应对

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| 附注格式多样，正则覆盖率低 | 中 | 中 | 对无法解析的附注整块保留，供 RAG 检索；后期引入小模型分类 |
| akshare 数据不稳定或延迟 | 低 | 中 | 以 PDF 解析为主，akshare 仅作校验；超时则跳过校验 |
| 跨年 RAG 检索召回精度不足 | 中 | 低 | 允许用户手动指定要检索的年份；同时提供全文搜索 fallback |
| token 成本仍然较高 | 低 | 中 | 通过优先级过滤 + 结构化指标预计算，将上下文压缩至 3000 tokens 以内 |
| 规则引擎无法覆盖所有造假手法 | 中 | 高 | 规则作为第一道过滤，LLM 负责发现未知异常；持续更新规则库 |

## 七、总结

本方案以现有 **a-share-financial-forensic** skill 为核心分析引擎，通过前置的结构化抽取、附注对齐、第三方验证和知识库建设，解决了原始 PDF 质量、token 成本、跨年分析、多源数据融合等关键问题。实施后，系统将具备：

- **高准确率**：关键指标双重验证，附注与报表自动对齐。
- **低成本**：80% 的规则判断由本地代码完成，LLM 仅用于推理和解释。
- **可扩展**：新增公司只需运行一次预处理流水线，即可纳入知识库。
- **产品化就绪**：可提供 API 或对话界面。

请团队按 Phase 0 → 5 逐步推进，每阶段完成后进行验收测试。如有技术选型或优先级调整，请及时反馈。
